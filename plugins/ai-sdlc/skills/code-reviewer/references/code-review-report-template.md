## Severity Level Definitions

### Critical

**When to Use**:
- Security vulnerabilities that could lead to:
  - Data breaches
  - Unauthorized access
  - System compromise
  - Data loss
- Production-breaking bugs
- Performance issues causing system failure

**Characteristics**:
- **Must** be fixed before production
- **Blocks** deployment
- High severity impact
- Immediate attention required

**Examples**:
- Hardcoded API keys or passwords
- SQL injection vulnerabilities
- Missing authentication on sensitive endpoints
- Memory leaks causing crashes
- Data corruption risks

---

### Warning

**When to Use**:
- Code quality issues affecting maintainability
- Performance problems affecting user experience
- Potential bugs that don't immediately break the system
- Best practice violations
- Security issues with lower risk

**Characteristics**:
- **Should** be fixed before merge
- May proceed with documented acceptance
- Medium severity impact
- Fix within current sprint

**Examples**:
- N+1 queries (scale dependent)
- High complexity functions (>20 complexity)
- Missing error handling
- Code duplication
- Missing indexes on frequently queried columns
- Potential XSS with sanitization

---

### Info

**When to Use**:
- Code smells
- Minor quality improvements
- Style consistency
- Documentation gaps
- Optimization opportunities

**Characteristics**:
- **Nice to fix** but not blocking
- Can be addressed later
- Low severity impact
- Add to backlog

**Examples**:
- TODO comments
- Magic numbers
- Minor code duplication
- Naming improvements
- Documentation additions
- Minor performance optimizations

## Warnings

### 1. Performance - Potential N+1 Query

**Location**: `src/features/payment/services/order.service.ts:78`
**Severity**: Warning
**Category**: Performance

**Issue**:
Database query inside map function could result in N+1 query problem when processing multiple orders.

**Code**:
```typescript
async getOrdersWithCustomers(orderIds: string[]) {
  const orders = await Order.findAll({ where: { id: orderIds } });

  return Promise.all(orders.map(async (order) => {
    const customer = await Customer.findByPk(order.customerId); // N queries
    return { ...order, customer };
  }));
}
```

**Impact**:
For 100 orders, this creates 101 database queries instead of 2. Performance degrades linearly with order count.

**Recommendation**:
Use eager loading or batch fetch customers.

**Example Fix**:
```typescript
async getOrdersWithCustomers(orderIds: string[]) {
  const orders = await Order.findAll({
    where: { id: orderIds },
    include: [Customer] // Eager load - 1 query with JOIN
  });

  return orders;
}
```

### 1. [SECURITY] Hardcoded JWT Secret

**Location**: `src/features/authentication/config/jwt.ts:5`
**Severity**: Critical
**Category**: Security
**CWE**: CWE-798 (Use of Hard-coded Credentials)

**Vulnerability**:
JWT secret key is hardcoded in source code, exposing it in version control and making all tokens vulnerable.

**Vulnerable Code**:
```typescript
export const jwtConfig = {
  secret: 'my-super-secret-jwt-key-12345',
  expiresIn: '24h'

```

**Attack Scenario**:
1. Attacker finds secret in git history or source code
2. Attacker generates valid JWT tokens for any user
3. Full authentication bypass - attacker gains access as any user

**Impact**:
- Complete authentication bypass
- Unauthorized access to all user accounts
- Potential data breach

**Recommendation**:
Move secret to environment variable immediately.

**Secure Implementation**:
```typescript
export const jwtConfig = {
  secret: process.env.JWT_SECRET,
  expiresIn: '24h'
};

// Add validation at startup
if (!process.env.JWT_SECRET) {
  throw new Error('JWT_SECRET environment variable is required');
}
```

**Priority**: Immediate - This is a critical security vulnerability

### 2. [SECURITY] SQL Injection in Login

**Location**: `src/features/authentication/services/auth.service.ts:45`
**Severity**: Critical
**Category**: Security
**CWE**: CWE-89 (SQL Injection)

**Vulnerability**:
User-provided email concatenated directly into SQL query without parameterization.

**Vulnerable Code**:
```typescript
async login(email: string, password: string) {
  const query = `SELECT * FROM users WHERE email = '${email}'`;
  const user = await db.query(query);

  if (user && await bcrypt.compare(password, user.password)) {
    return generateToken(user);
  }
  throw new Error('Invalid credentials');
}
```

**Attack Scenario**:
Email input: `admin@example.com' OR '1'='1' --`
Resulting query: `SELECT * FROM users WHERE email = 'admin@example.com' OR '1'='1' --'`

This bypasses authentication and returns first user (often admin).

**Impact**:
- Authentication bypass
- Unauthorized access
- Potential data exfiltration

**Recommendation**:
Use parameterized queries.

**Secure Implementation**:
```typescript
async login(email: string, password: string) {
  const user = await db.query(
    'SELECT * FROM users WHERE email = ?',
    [email] // Parameterized - safe from injection
  );

  if (user && await bcrypt.compare(password, user.password)) {
    return generateToken(user);
  }
  throw new Error('Invalid credentials');
}
```

**Priority**: Immediate - Critical authentication vulnerability

### 3. [SECURITY] Missing Authorization Check

**Location**: `src/features/authentication/api/users.ts:78`
**Severity**: Critical
**Category**: Security
**CWE**: CWE-862 (Missing Authorization)

**Vulnerability**:
User update endpoint has authentication but no authorization check - any authenticated user can modify any user's profile.

**Vulnerable Code**:
```typescript
app.put('/api/users/:id', requireAuth, async (req, res) => {
  // No check if req.user.id === req.params.id
  await User.update(req.params.id, req.body);
  res.json({ success: true });
});
```

**Attack Scenario**:
1. User A (ID: 123) is authenticated
2. User A sends: `PUT /api/users/456` with admin role change
3. User A successfully elevates User B (ID: 456) to admin
4. Privilege escalation attack

**Impact**:
- Unauthorized profile modifications
- Privilege escalation
- Data integrity violation

**Recommendation**:
Add ownership verification.

**Secure Implementation**:
```typescript
app.put('/api/users/:id', requireAuth, async (req, res) => {
  const targetUserId = req.params.id;
  const currentUserId = req.user.id;

  // Only allow users to update their own profile (unless admin)
  if (targetUserId !== currentUserId && !req.user.isAdmin) {
    return res.status(403).json({
      error: 'Forbidden: Cannot modify other users'
    });
  }

  // Prevent non-admins from changing role
  if (req.body.role && !req.user.isAdmin) {
    return res.status(403).json({
      error: 'Forbidden: Cannot change role'
    });
  }

  await User.update(targetUserId, req.body);
  res.json({ success: true });
});
```

**Priority**: Immediate - Authorization bypass vulnerability

## Usage Guidelines

### When Writing Reports

1. **Be Specific**: Always include file paths, line numbers, and code snippets
2. **Be Clear**: Explain why something is an issue, not just that it is
3. **Be Actionable**: Provide concrete fix examples, not just "fix this"
4. **Be Accurate**: Verify findings to avoid false positives
5. **Be Contextual**: Consider project-specific standards and constraints

### Severity Assignment

**Critical** = Must fix now (blocks deployment)
**Warning** = Should fix soon (blocks merge)
**Info** = Nice to fix (backlog)

When in doubt, prefer lower severity. Better to under-report than over-report.

### Report Customization

**For Security-Only Scans**: Omit quality and performance metrics sections

**For Quick Scans**: Reduce detail in informational section, focus on critical/warnings

**For Large Codebases**: Limit to top 20 issues per severity level to keep report manageable

## Summary

This template ensures consistent, comprehensive, and actionable code review reports. Always provide clear severity levels, specific locations, concrete examples, and actionable recommendations. The goal is to help developers improve code quality efficiently.
